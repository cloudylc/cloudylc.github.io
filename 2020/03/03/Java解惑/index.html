    01 奇数的判定

    ​ 当你需要判定一个数是否为奇数时， 我想你大概会想到以下代码，但那是有错误的！

    public static boolean isOdd(int i){    
    return i % 2 == 1;
}

    ​ 但我们仔细查看关于奇数的定义：

    奇数（odd）指不能被2整除的整数 ，数学表达形式为：2k+1， 奇数可以分为正奇数和负奇数。

    以上代码的错误，就是没考虑到判定数为负数的情况，所以正确的奇数算法如下：

    public static boolean isOdd(int i){    
    return i % 2 != 0;
}

    ​ 当需要强调性能需求时，用位操作符AND（&）代替取余操作符会显得更好。

    public static boolean isOdd(int i){    
    return (i & 1) != 0;
}

    02 (2.0 - 1.1)的结果

    ​ 观察以下代码，说出你的答案。

    public calss Change{
    public static void main(String[] args){
        System.out.printf(2.0 - 1.1);
    }
}

    ​ 你可能以为答案是0.9，但你运行此程序，打印的结果确是0.8999999999999999。是不是觉得很奇怪？

    ​ 问题就出在1.1这个数并不能被精确的表示为一个double，而是被表示为最接近它的double值。更一般的说，计算机在对于浮点数的表示并非都是精确的，并非所有的小数都可以用二进制浮点数精确表示。

    ​ 以下是一个JDK 5.0或更新版本的陷阱，通过使用printf工具设置输出精度的方法更正该程序。但很遗憾的是，虽然显示的是正确的结果，但对于底层的实现仍旧是使用二进制浮点数的double运算。浮点运算在一个范围很广的值域上提供了很好的近似值，但通常不能产生精确的结果。对于实际开发而言，二进制浮点数对于货币计算是非常不适合的，因为二进制浮点数不可能将0.1或者10的任何次负幂，精确表示为一个有限长度的二进制小数。

    System.out.printf("%.2f%n",2.00 - 1.10);

    ​ 解决的方法有两个，一个是把小数提升到整数运算，另外一个方式则是使用执行精确小数运算的BigDecimal。对于第一种解决方式，把1.1RMB变成110角，由float或double提升到int或long，但该方法需要确保该整数类型大到足以表示程序中将要用到的所有值。

    ​ 第二种解决方式，它还可以通过JDBC与SQL DECIMAL类型进行互操作。这里要告诫一点：一定要使用BigDecimal(String)构造器，千万不能使用BigDecimal(Double)。因为后者会使用它参数的精确值来创建一个实例，返回一个很奇怪的BigDecimal。

    BigDecimal aa = new BigDecimal(0.1);
BigDecimal cc = new BigDecimal("0.1");
// aa = 0.1000000000000000055511151231257827021181583404541015625
// bb = 0.1

    ​ 以下是正确使用BigDecimal来得到我们期望的0.90。但这个版本并非完美，因为Java并没有给BigDecimal提供任何语言上的帮助，在运行速度上可能比不过原生类型，对于大量使用小数计算的程序而言，这可能会成为问题，而对于大多数程序来说，这点显得一点也不重要。

    import java.math.BigDecimal;
public class isOdd {
    public static void main(String[] args) {
        BigDecimal aa = new BigDecimal("2.0");
        BigDecimal cc = new BigDecimal("1.1");
        System.out.println(aa.subtract(cc));
    }
}

    ​ 以上我们得到3方面的提示，在需要精确答案的地方，要避免使用float和double；对于货币计算，要使用int、long或BigDecimal。对于语言设计者来说，应该考虑对小数运算提供语言支持。
